## 1. ゴール定義

以下がすべて満たされた状態を「完成」とする。

1. 16パックすべてについて、3枚のHTML素材（waiting / brb / lower_third）がChromeとOBS Studioのブラウザソースで表示エラーなく動作する（JSエラー0件、レイアウト崩れなし）。
2. 16パックすべてについて、BOOTH商品画像用スクリーンショットPNGが各3枚以上（waiting全景・brb全景・lower_third合成例）、1920x1080で `booth/thumbnails/<部屋ID>/` に存在する。
3. 16パックすべてについて、BOOTH商品ページ用テキスト（タイトル・説明文・タグ・価格）が `booth/listings/<部屋ID>.md` として存在する。
4. シリーズ共通のブランド要素（シリーズロゴ入りサムネイルテンプレート、シリーズ説明文、既存商品との相互リンク文）が `booth/listings/_series.md` として存在する。
5. まとめ買い用の全部屋バンドルzip（`KYOUKAI_全部屋_OBS素材パック.zip`）が生成されている。
6. BOOTHへの出品操作は**ユーザー（まろ）が手動で行う**。出品自体はゴールに含めないが、出品作業がコピペだけで完了する状態のチェックリスト `booth/listings/_出品手順.md` を用意する。

曖昧成功条件の禁止事項: 「いい感じのサムネ」等は不可。サムネイルは上記の解像度・枚数・保存場所で判定する。

## 2. 全体アーキテクチャ

### 採用技術

| 要素 | 技術 | 理由 |
|---|---|---|
| 素材本体 | 静的HTML+CSS+Canvas（生成済み） | OBSブラウザソースで完全ローカル動作。外部依存ゼロが商品価値 |
| パック生成 | `booth/generate_packs.py`（既存） | 16部屋のデータ駆動生成。修正はこのスクリプト経由で行い、生成物を直接編集しない |
| スクリーンショット | Playwright (Python) | `shorts_factory/` で導入済みの実績あり。1920x1080固定・ヘッドレスで再現性が高い |
| サムネ文字入れ | Pillow (PIL) | ローカル完結。シリーズロゴ・部屋名の重ね描き |
| 検証 | Playwrightでのconsoleエラー収集 | 16×3=48ファイルの目視確認は非現実的。自動化必須 |

### 却下した選択肢

- **Chrome拡張(claude-in-chrome)での手動スクショ**: 前セッションで実施したが、ウィンドウサイズ依存・スケール補正が必要で16パック×3枚=48枚には非現実的。却下。
- **html2canvasでのブラウザ内キャプチャ**: Canvas要素のtaint問題とアニメーションタイミング依存があり品質が不安定。却下。
- **WebM動画素材化**: 差別化にはなるがエンコード工程・ファイルサイズ・品質調整の工数が大きい。v2で検討。今回は却下。
- **BOOTH API自動出品**: BOOTHに公開APIが存在しない。出品は手動と割り切る。

### ディレクトリ設計（今回追加分）

```
booth/
  all-packs/           # 生成済み（既存）
  generate_packs.py    # 既存ジェネレーター
  screenshot_packs.py  # 工程2で新規作成
  make_thumbnails.py   # 工程3で新規作成
  thumbnails/<部屋ID>/ # 工程2-3の出力
  listings/            # 工程4-5の出力
    _series.md
    _出品手順.md
    <部屋ID>.md × 16
```

部屋IDは `generate_packs.py` の `ROOMS` の `id` フィールド（observation, hyougi, null, observer, exit, archive, ma, daimyojin, gokuraku, particles, ripple, kanrinin, namahage, matsuri, fukashitsu, typhoon-news）を使用する。

## 3. 工程分割（全6工程）

### 工程1: 全パックHTML品質検証と修正

状態: 完了

**目的**: 生成済み48ファイル（16部屋×3種）がエラーなく動作することを自動検証し、問題があれば `generate_packs.py` を修正して再生成する。

**実装内容**:
- `booth/verify_packs.py` を新規作成。
- Playwright(chromium, headless)で各HTMLを `file://` で開き、以下を収集:
  - `page.on('console')` で `error` レベルのメッセージ
  - `page.on('pageerror')` で未捕捉例外
  - 3秒待機後の `canvas` 要素の存在確認（waiting/brbのみ）
  - lower_third: `#name-el` のtextContentが「名前」であること（URLパラメータなし時）
- 結果を `booth/verify_report.json` に部屋ID×ファイル別で出力。
- エラーが出た部屋は `generate_packs.py` の該当データ/テンプレートを修正し、`python generate_packs.py` で全再生成してから再検証する。生成物のHTMLを直接編集しない。
- Playwright未導入なら `pip install playwright && playwright install chromium` を実行する。

**完了条件**:
- [x] `booth/verify_packs.py` が存在し、48ファイル全件を検証する
- [x] `booth/verify_report.json` の全エントリが pass である
- [x] エラー修正があった場合、`generate_packs.py` 修正→再生成→zipの再作成まで完了している（今回は全件passのため修正不要）

**落とし穴**:
- `file://` でのフォント読み込みは問題ないが、パスに日本語が含まれるためPlaywrightへは `Path.as_uri()` で渡すこと（手組みURLはエンコード事故のもと）。
- Canvas系スタイル（particles等）はrequestAnimationFrameが重い可能性。エラーではなくパフォーマンス警告は無視してよい。
- cp932問題: print出力に✓や—を使わない（前回発生済み）。

**推奨実装モデル**: sonnet

**完了条件**:

- [x] すべて1920x1080である（PillowでサイズをassertするチェックをスクリプトP末尾に含める）
- [x] waiting系スクショにCanvasアニメーションが写っている（真っ黒でない。ピクセル平均輝度>1で判定）

**落とし穴**:

## 5. リスク一覧

| リスク | 影響 | 対策 |
|---|---|---|
| BOOTHの規約変更・審査で説明文の演出が引っかかる | 出品差し戻し | 実用情報（内容物・動作環境・規約）を必ず明記。演出は冒頭のみ |

| 16商品出しても売れない | 工数損 | 素材はデータ駆動生成のため追加工数は小さい。1つでも売れればKYOUKAI本体への導線になる |
| 類似商品との差別化不足 | 埋没 | 「16部屋の世界観シリーズ」であること自体が差別化。商品ページ相互リンクで回遊させる |
